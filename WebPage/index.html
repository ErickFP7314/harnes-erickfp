ErickFP

Harness agéntico de línea de comandos que gobierna a cualquier modelo con el método cartesiano: primero duda, luego divide, después construye y al final revisa — con trazabilidad ADR y un gate de permisos que nunca se salta.

Instalar ahora Leer la guía

La idea

"El modelo es el motor, el harness es todo lo demás." ErickFP no compite por tener el modelo más listo: compite por darle a cualquier modelo (incluso uno gratuito) una disciplina de trabajo verificable — el Ciclo Cogito, inspirado en el método de Descartes: duda de la ambigüedad, divide el problema en partes simples, ordena la construcción de lo simple a lo complejo y enumera (revisa) el resultado antes de darlo por bueno.

Diferenciadores frente a otros harnesses

  • Método cartesiano gobernando el flujo. No es una sugerencia de prompt: duda → divide → ordena → enumera son fases bloqueantes, en orden obligatorio, cada una validando la anterior.
  • Trazabilidad ADR obligatoria. Un grafo de decisiones de arquitectura enlazadas por frontmatter YAML; un hook valida por DFS que toda síntesis trace hasta un nodo raíz — ninguna decisión queda huérfana.
  • Gate de permisos default-deny + núcleo sagrado. Toda tool pide y/n y solo y exacto aprueba; además .ErickFP/core/ está protegido por un hook que bloquea a la IA incluso si el humano aprobó por error.
  • Provider-agnostic verificado por test. Interfaz Provider propia; un test AST prohíbe importar SDKs nativos fuera del adapter — cambiar de modelo no reescribe el harness.
  • Free tier por defecto. Corre gratis sobre el free tier de Gemini (sin tarjeta de crédito para empezar).

Cómo funciona: el Ciclo Cogito

Cuatro fases secuenciales, cada una con su propio comando y su propio artefacto en .ErickFP/cogito/<slug>/.

  1. 1. Duda Evidencia

    Valida que el objetivo no sea ambiguo. Si lo es, el harness se niega a avanzar y pregunta.

    erickfp duda "<objetivo>"
  2. 2. Divide Análisis

    El rol Planner descompone el objetivo en las partes más pequeñas posibles.

    erickfp divide <slug>
  3. 3. Ordena Síntesis

    El rol Coder construye de lo simple a lo complejo, trazable al grafo ADR.

    erickfp ordena <slug>
  4. 4. Enumera Revisión

    El rol Reviewer repasa todo lo producido antes de darlo por cerrado.

    erickfp enumera <slug>

Features

Ciclo Cogito

4 fases cartesianas (duda/divide/ordena/enumera) con artefactos encadenados y roles Planner/Coder/Reviewer.

Permission gate sin fuga

Toda tool pide y/n; solo y exacto aprueba. Default deny.

PermissionPolicy pluggable

AlwaysAsk (default), AllowList, AskOnce.

Núcleo sagrado

.ErickFP/core/ protegido por hook: ni el humano aprobando puede saltárselo.

Grafo ADR trazable

Toda síntesis debe trazar por DFS hasta un nodo raíz del grafo de decisiones.

Hooks por fase

PreToolUse/PostToolUse/PhaseStart/PhaseEnd, acumulativos y extensibles.

Provider-agnostic

Interfaz propia; único import de LiteLLM aislado en el adapter; SDKs nativos prohibidos por test.

Tools + MCP

bash, read_file, write_file, recall, delegate — y tools remotas vía MCP tras la misma interfaz.

Memoria persistente

SQLite save/recall/preamble acotado + resumen de sesión.

Compaction

SlidingWindow/Summarize + SafeSplitPoint: nunca corta un tool_use/tool_result.

Subagentes

Clase Agent reutilizable + subagente Research read-only + DelegateTool.

Token viewer

/tokens: uso y costo estimado por sesión, en tiempo real.

UI terminal

Banner ASCII 25×149 precomputado, input decorado, tema cyan/verde sobre #222222.

Calidad

TDD estricto + arquitectura de capas verificada por import-linter.
